# FOLLOW ENGINE SPEC — `[UNBUILT]`

> 🔴 **`[UNBUILT]` — feature นี้ยังไม่สร้าง** · โค้ดปัจจุบันมีแค่ `follow_log` dedup + cron "เงียบเกิน N วัน" (dormant · สวิตช์ `เปิด_ระบบติดตาม` ปิด) · `customer_tags`/`follow_queue`/tag-triggered engine **ไม่มีในโค้ดเลย**
> เอกสารนี้ = **design เต็มของระบบ follow ใหม่ (tag-triggered)** ที่ตั้งใจ replace ของ dormant ปัจจุบัน · พอ build เสร็จ → ดูดสรุปเข้า REPO-MAP แล้วลบไฟล์นี้
> ระบบติดตามลูกค้าแบบ **tag-triggered** — "ไม่มีแท็ก = ไม่ตาม · ขั้นเลื่อนเมื่อส่งจริง · ลูกค้าทัก = ยกเลิกคิวไม่รีเซ็ตขั้น"

---

## 1. หลักการ (จากเจ้าของ)

> **ไม่มีแท็ก = ไม่ตาม** แม้เปิดสวิตช์แล้วก็ตาม
> **ขั้นเลื่อนก็ต่อเมื่อข้อความตามถูกส่งจริง** เท่านั้น
> **ลูกค้าทักมา = ยกเลิกคิวที่ตั้งไว้ แต่ไม่รีเซ็ตขั้น**
> **เริ่มนับใหม่ก็ต่อเมื่อเปลี่ยน track** (สนใจ → รอโอน)

**ตัวอย่างที่ต้องทำงานถูก:**
- ลูกค้า A สนใจ → เงียบ → ติด `ตาม:NPT:1`
- ส่ง ตาม1 แล้ว → แท็กเปลี่ยนเป็น `ตาม:NPT:2` (ขั้นเลื่อน **เพราะส่งจริง**)
- ลูกค้าทักมาถามนู่นนี่ แต่ยังไม่สั่ง → **ยกเลิกคิว แต่แท็กยังเป็น `ตาม:NPT:2`** → รอบหน้าได้ ตาม2 ไม่ใช่ ตาม1 ซ้ำ
- ลูกค้าสั่ง → สรุปยอด → **เปลี่ยน track**: ลบ `ตาม:NPT:*` ทิ้ง → ติด `รอโอน:1`
- ตามโอน1 แล้ว ลูกค้าบอก "เดี๋ยวพรุ่งนี้โอน" แล้วหาย → แท็กเป็น `รอโอน:2` อยู่แล้ว → รอบหน้าได้ ตามโอน2 (ไม่ใช่ ตามโอน1 ซ้ำ) ✅

## 2. ทำไมต้องเปลี่ยน schema ก่อน

```
customers.tags TEXT[]   ← มีแค่ชื่อแท็ก ไม่มีว่าติดเมื่อไหร่
```
ทั้งระบบตั้งอยู่บน *"นับจากวันที่ติดแท็กไปอีก 3 วัน"* — แต่ข้อมูลนั้น **ไม่มีอยู่จริง**
เพิ่มคอลัมน์ในชีต Follow ยังไงก็ไม่ช่วย

```sql
customer_tags(user_id, tag, applied_at, source, PRIMARY KEY(user_id, tag))

follow_queue(
  id BIGSERIAL PK,
  user_id, rule_name, trigger_tag,
  tag_applied_at TIMESTAMPTZ,
  due_at TIMESTAMPTZ,
  status TEXT,              -- pending | sent | cancelled | skipped
  sent_at, cancel_reason,
  UNIQUE(user_id, rule_name, tag_applied_at)
)
```

**ทำไมต้องมี `follow_queue` (ไม่คำนวณสดใน cron):**
1. **เทสได้** — ไม่ต้องรอ 3 วัน แค่ดูว่า `due_at` ถูกไหม
2. **ยกเลิกได้** — ลูกค้าทักมา = `UPDATE status='cancelled'` จบ
3. **เห็นค่าใช้จ่ายล่วงหน้า** — query ได้ว่าพรุ่งนี้ 10:00 จะยิงกี่ push (= เงินจริง)
4. **กันส่งซ้ำ** — `UNIQUE(user_id, rule_name, tag_applied_at)`

## 3. ชีต CSV_Follow ใหม่ (header-driven)

| header | ความหมาย |
|---|---|
| `ชื่อกฎ` | unique · key ของ follow_log |
| `แท็กที่ trigger` | **หัวใจ** — ไม่มีแท็กนี้ = กฎนี้ไม่ทำงาน · รองรับ `{sku}` |
| `รอ_จำนวน` | ตัวเลข |
| `รอ_หน่วย` | `วัน` \| `ชม.` |
| `ส่งเวลา` | `10:00` · **ว่าง = ส่งตาม due_at ตรงๆ** (ใช้กับกฎหน่วยชั่วโมง) |
| `ถ้าเลยเวลา` | `19:00` · ครบกำหนดหลังเวลานี้ → เลื่อนไป `ส่งเวลา` ของวันถัดไป |
| `ข้อความ (บอลลูน)` | รองรับ `\n` และ `[[แยก]]` |
| `ลบแท็ก` | หลังส่ง**สำเร็จ**เท่านั้น |
| `เพิ่มแท็ก` | หลังส่ง**สำเร็จ**เท่านั้น → trigger กฎถัดไปเอง |
| `ลำดับความสำคัญ` | เลขน้อย = สำคัญกว่า (รอโอน=10 · รอที่อยู่=20 · สนใจ=30) |
| `เปิด/ปิด` | ปิดกฎรายตัวได้ ไม่ต้องลบแถว |

**ไม่มีคอลัมน์ "หยุดตามเมื่อ" อีกแล้ว** — "ลูกค้าตอบกลับ = ยกเลิกคิว" เป็นกฎของ **ระบบ** ไม่ใช่กฎรายแถว
การให้โค้ดตีความข้อความไทยอิสระคือทางที่พังแน่นอน (บทเรียนจาก v1.1 ที่คอลัมน์ B/C/F/G ไม่เคยถูกอ่านเลย)

## 5. คำนวณ `due_at` (ทำครั้งเดียวตอนติดแท็ก)

```
1. base = tag_applied_at + (รอ_จำนวน × รอ_หน่วย)
2. ถ้า `ส่งเวลา` ไม่ว่าง → ปัดไปที่เวลานั้นของวันเดียวกัน
3. ถ้า base เลย `ถ้าเลยเวลา` แล้ว → เลื่อนไป `ส่งเวลา` ของวันถัดไป
4. ถ้า due_at ตกในช่วงห้ามส่ง (ตาม_เวลาห้ามส่ง_เริ่ม 19:00 → ตาม_เวลาห้ามส่ง_จบ 08:00)
   → เลื่อนไป `ส่งเวลา` ของวันถัดไป
5. เวลาไทย (UTC+7) เสมอ
```
**ตัวอย่าง:** ติดแท็ก `รอโอน:1` เวลา 18:30 · กฎ = รอ 2 ชม. · ถ้าเลยเวลา 19:00
→ base = 20:30 → อยู่ในช่วงห้ามส่ง → เลื่อนเป็น **พรุ่งนี้ 10:00** ✅ (ตรงตามที่เจ้าของอธิบาย)

## 6. Lifecycle hooks (4 จุด — พลาดจุดไหนระบบพัง)

### H-1 · ติดแท็ก (โค้ดตัดสิน ไม่ใช่ AI)
```
addTag(user, tag) →
  INSERT customer_tags(user_id, tag, applied_at=now)
  หา rule ในชีต Follow ที่ trigger_tag ตรง + เปิดอยู่
  → INSERT follow_queue(status='pending', due_at=คำนวณจากข้อ 5)
```

### H-2 · ลูกค้าทักมา (write path) — **ยกเลิก แต่ไม่รีเซ็ตขั้น**
```
ทุก inbound message (ก่อนบอทตอบ):
  UPDATE follow_queue SET status='cancelled', cancel_reason='customer_replied'
    WHERE user_id=? AND status='pending'
  ⚠️ ไม่แตะ customer_tags — แท็กและขั้นคงเดิม
```

### H-3 · จบเทิร์น → ประเมิน track ใหม่ (โค้ดตัดสิน)
```
track ปัจจุบัน = จาก pending_order + สถานะออเดอร์
  ถ้า track เดิม (ยังแค่สนใจ)      → คงแท็กเดิม + re-arm คิวใหม่จาก now (ขั้นเดิม)
  ถ้า track เปลี่ยน (สนใจ→รอโอน)   → ลบแท็ก track เดิมทั้งหมด + ติดแท็ก track ใหม่ที่ขั้น 1
  ถ้าปิดการขาย / handoff / human_mode → ลบแท็ก follow ทั้งหมด + cancel คิวทั้งหมด
```
> **นี่คือกฎที่เจ้าของย้ำ:** ลูกค้าคุยต่อแต่ยังไม่เปลี่ยนสถานะ = ตามข้อความ**ถัดไป** ไม่ใช่เริ่ม 1 ใหม่

### H-4 · cron จะส่ง (send path) — **guard ชั้นที่ 2 กัน race**
```
ก่อน push ทุกครั้ง อ่าน state สดอีกรอบ · ข้ามถ้า:
  - last_seen > tag_applied_at        (ลูกค้าทักมาหลังติดแท็ก)
  - trigger_tag หายไปจาก customer_tags แล้ว
  - human_mode = true
  - เกิน `ตาม_เพดานต่อคน_ครั้ง` ใน 30 วัน (นับจาก follow_log)
  - ส่ง follow ให้คนนี้ไปแล้ววันนี้ (`ตาม_สูงสุดต่อคนต่อวัน`)
  - อยู่ในช่วงห้ามส่ง
ส่งสำเร็จ → status='sent' + INSERT follow_log
          + ลบแท็ก (`ลบแท็ก`) + ติดแท็กใหม่ (`เพิ่มแท็ก`) ← ขั้นเลื่อนตรงนี้ที่เดียว
          + แท็กใหม่ trigger กฎถัดไปอัตโนมัติผ่าน H-1
```
**H-2 อย่างเดียวไม่พอ** — cron อาจกำลังรันอยู่พอดีตอนลูกค้าพิมพ์ H-4 คือตัวกัน race จริง

## 7. กติกากันพัง (ตั้งใน Config)

| key | ค่า | กัน |
|---|---|---|
| `ตาม_เพดานต่อคน_ครั้ง` | 7 | **ลูปไม่รู้จบ** — ลูกค้าตอบ "ค่ะ" ทุกครั้งแล้วหาย → ตามตลอดชีพ · เพดานนี้ **ไม่รีเซ็ตตอนลูกค้าตอบ** |
| `ตาม_สูงสุดต่อคนต่อวัน` | 1 | ลูกค้าสนใจ 2 SKU + รอโอน → 10:00 ได้ 3 ข้อความรวด = ประสบการณ์พัง · เลือกตาม `ลำดับความสำคัญ` ที่เหลือเลื่อน |
| `ตาม_เวลาห้ามส่ง_เริ่ม/จบ` | 19:00 / 08:00 | ยิงตอนดึก = โดนบล็อก |
| `ตาม_รอบ cron` | 15 นาที | 🔴 **cron ปัจจุบันวันละครั้ง (Vercel Hobby) → กฎหน่วย "2 ชั่วโมง" ทำไม่ได้** ต้องย้ายไป external cron (cron-job.org) เหมือน `/api/cron/orders` |

**`ตามครบ:{track}`** = แท็กปลายทาง ไม่มีกฎ trigger ต่อ = หยุดตาม ลูกค้ากลุ่มนี้ไปอยู่ broadcast pool (คนละระบบ ไม่ใช่ follow)

## 8. เตรียมไว้สำหรับเฟส 3 (ทำตอนนี้ ฟรี)

เพิ่มคอลัมน์ `follow_variant` ใน `follow_queue` — สุ่ม A/B ข้อความตามตั้งแต่วันแรก
คำถาม *"follow แบบไหนดี"* ตอบได้ก็ต่อเมื่อ**สุ่ม**เท่านั้น ถ้าไม่สุ่มจะได้แค่ correlation ที่หลอกตัวเอง
ทำวันนี้ = 1 คอลัมน์ · ทำทีหลัง = ข้อมูล 3 เดือนแรกใช้ไม่ได้
